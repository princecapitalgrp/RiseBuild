Personalization Engine — Service Boundaries

InterpretationService
Input: RawCheckIn + OnboardingProfile
Output: DailyContext (Layer 2)
Where: On-device only
Rule: No user-authored text in output
Future: On-device LLM for emotional nuance
RoutineGenerationService
Input: DailyContext + MemoryContext + Profile
Output: Protocol (Layer 3)
Where: /api/generate-protocol (server)
Key: OpenAI key stays server-side
Fallback: Rule-based engine on error
ExecutionTracker
Input: ExecutionEvent (any point in session)
Output: Appends to ExecutionRecord
Where: MMKV, synchronous
Rule: Append-only. Events are never deleted.
MemoryEngine
Input: Protocol + ExecutionRecord + Reflection
Output: Updated MemorySummary
Where: On-device
Also: getContextSummary() → patterns only, for AI
FitEvaluator — Pure Computation
Inputs: Protocol + ExecutionRecord + Reflection
Output: FitScore (not persisted — computed on read)
No side effects. Can be called any time.
FitScore components:
userRated (feltAccurate normalized)
executionDepth (steps completed %)
followThrough (yes/partial/no)
timingAccuracy (prescribed vs actual)

5 Metrics — Is the PE Getting Better?

1
Rolling FitScore Trend
Composite score averaged over 7 days. Target: week 4 measurably above week 1. Primary health signal.
2
Execution Depth Trend
% of prescribed steps executed. Rising trend = protocols are being built at the right complexity for this user.
3
Follow-Through Rate
Frequency of followedIt: 'yes' over time. Increasing = protocols feel achievable, not aspirational.
4
User-Rated Accuracy Trend
feltAccurate rolling average. The self-reported signal that the app understands the user unusually well.
5
High-Accuracy Pattern Density
Rate at which highAccuracyPatterns accumulate vs lowAccuracyPatterns in MemorySummary. The PE learning signal.
All five metrics are computed on-device from MMKV data. No data leaves the device to generate them. Aggregate (anonymized) versions of metrics 1–3 can optionally be surfaced to Firestore analytics if user opts in.
